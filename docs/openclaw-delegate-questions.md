# OpenClaw / 主·子 Agent 相关疑惑记录

记录产品与设计上的开放问题，供后续迭代参考。

---

## 1. 子 agent 可否由主 agent 动态创建？

**现状**：子角色是预先配置的（`gateway/data/agents/<role>.md`，或兼容 `subpersona/`），主 agent 只能在回复里写 `DELEGATE: 任务 | 已存在的角色名`。

**疑惑**：子 agent 能不能由主 agent **动态创建**，而不是写死的一套流程化内容？  
例如：主 agent 根据用户需求临时「定义」一个子角色（名字 + 一段 system 描述），再由网关按该描述拉起一次性子 agent。

**待讨论**：实现方式（动态 system 注入 vs 固定文件）、安全与上下文长度、是否保留「预置子角色 + 动态子角色」并存。

---

## 2. 主 agent 多任务依赖：A、B、C，且 C 依赖 A、B 的结果

**现状**：当前实现是「主回复 → 解析出多个 DELEGATE → **串行**执行子 agent（先 A 再 B 再 C）→ 子结果拼成一段 → 再调主 agent 综合」。没有表达「C 依赖 A、B」的依赖关系。

**疑惑**：若主 agent 有三个任务 A、B、C，且 **C 依赖 A 和 B 的结果**，应如何表达与执行？  
例如：先并行或串行执行 A、B，再把 A、B 的结果作为 C 的输入，最后综合。

**待讨论**：是否需要在 DELEGATE 格式或协议里表达依赖（如 `DELEGATE: 任务C | coder | 依赖:task_a,task_b`）；网关是否需要 DAG 调度（有向无环图）而非简单串行。

---

## 3. 派发任务是否并行？线性 vs 动态多线程

**现状**：多个 DELEGATE 时，**串行**执行（一个接一个调用子 agent），子结果用分隔符拼接后一次性回传主 agent。

**疑惑**：A 给 B、C 派发任务后，应该是 **并行** 执行还是保持线性？理解上希望是 **动态的、多线程/并行的**，以缩短总耗时。

**待讨论**：实现上改为 `Promise.all` 并行调用子 agent（无依赖时）；若有依赖（见上条），则需依赖感知的调度（部分并行 + 部分串行）。

---

## 4. 聊天执行任务时，OpenClaw 随时插入新任务并同时执行

**现状**：一次用户消息对应一条主回复 → 若有 DELEGATE 则串行子任务 → 再调主 agent 综合 → 返回。没有「用户或系统在任务执行过程中再插新任务、与当前任务并发」的机制。

**疑惑**：在触发聊天执行任务时，**OpenClaw 能否随时安排新任务**，与当前正在执行的任务 **同时执行**？  
例如：用户发了「查 REST API + 写示例」，子任务正在跑时，用户又发「再查一下 GraphQL」；或系统自动插入一个「保存进度」类任务。

**待讨论**：多请求/多会话与「当前任务队列」的关系、是否引入任务队列与取消/优先级、前端与网关的交互模型（单请求 vs 多请求并发、会话内任务 ID 等）。
